    React TDD Style Guide

    In React applications, the most common and effective TDD style is Chicago-style TDD, also known as Detroit-style. This approach focuses on testing logic, state, and visible behavior without relying heavily on mocks or testing internal implementation details.

    ✅ Most Common: Chicago-Style TDD

    🔍 Why it's popular in React

    Reason Explanation
    React favors pure logic Components and hooks are often functional and deterministic — perfect for input/output testing.
    Encouraged by React Testing Library RTL promotes testing what the user sees, not implementation details.
    Minimal mocking Most React logic doesn’t require mocks unless dealing with APIs or services.
    More maintainable tests Chicago-style tests are less brittle and easier to understand.

    🧪 Example: Chicago-Style React Test

    import { render, screen, fireEvent } from "@testing-library/react";
import LoginForm from "./LoginForm";

test("disables submit when fields are empty", () => {
  render(<LoginForm />);
  const button = screen.getByRole("button", { name: /login/i });
  expect(button).toBeDisabled();
});
    • ✅ Tests the behavior, not implementation
    • ✅ No mocking required
    • ✅ Reflects actual user interaction

    🧰 Recommended Tools for Chicago TDD in React

    • React Testing Library – For testing DOM behavior and interactions
    • Jest – For test running, assertions, and basic mocking
    • Vitest – (Optional) Faster alternative to Jest in Vite-based apps

    🤝 Other TDD Styles in React (Used Less Frequently)

    Style When It's Used
    London Style When isolating external services (e.g., mocking API or auth services)
    Component TDD When developing reusable, stylized UI components
    BDD / ATDD In end-to-end testing with Cypress, Playwright, or Gherkin
    Integration TDD When testing full flows (e.g., login → dashboard → profile)

    🧠 Summary

    • Chicago-style TDD is the dominant testing style in React for unit and component testing.
    • It aligns with how React encourages component composition and behavior-driven UI.
    • Use other styles (London, BDD, integration) selectively where appropriate.

    Test behavior, not implementation. Let the user’s experience guide your tests.

    Example

    In a React application, Chicago-style TDD focuses on testing the following key areas:

    ✅ Where Chicago-Style TDD Applies in React

    Layer What It Tests Example
    🧩 Components DOM output, state, props Button disabled/enabled, input value changes
    ⚙️ Hooks State transitions, return values Custom useCounter, useToggle
    🧠 Pure Functions Logic, formatting, validation formatDate(), calculateTax()

    🔍 1. Component Tests (Most Common)

    You test what the user sees and can interact with — without mocking internals.

    Example
    test("shows error if email is invalid", () => {
  render(<LoginForm />);
  fireEvent.change(screen.getByLabelText(/email/i), {
    target: { value: "invalid-email" },
  });
  fireEvent.click(screen.getByText(/submit/i));

  expect(screen.getByText(/invalid email/i)).toBeInTheDocument();
});
    • No mocks
    • Tests actual rendered output
    • Focus on visible behavior, not implementation

    🔁 2. Custom Hook Tests

    Hooks return values or functions — great for direct logic testing (Chicago style).

    import { renderHook, act } from "@testing-library/react-hooks";
import useCounter from "./useCounter";

test("increments counter", () => {
  const { result } = renderHook(() => useCounter());
  act(() => result.current.increment());
  expect(result.current.count).toBe(1);
});
    • Tests state transitions
    • No need to mock anything
    • Treats the hook like a pure function

    📐 3. Utility / Logic Functions

    Chicago-style is perfect for testing small helper logic:

    import { formatName } from "./utils";

test("formats name correctly", () => {
  expect(formatName("john", "doe")).toBe("John Doe");
});

    🧠 What It Avoids

    • ❌ Mocking internal props/state
    • ❌ Testing private methods or component internals
    • ❌ Over-isolation of components

    Instead, it encourages simple tests that reflect what the user would see or do.

    🔄 When It Mixes with Other Styles

    Style When You Might Use It in React
    London Style Mocking APIs or services (e.g., AuthService)
    BDD/ATDD Cypress or Playwright for user flows
    Integration TDD Testing multi-component flows

    📌 Summary

    In React, Chicago-style TDD is best used to test:

    • ✅ Component rendering and behavior
    • ✅ Hook logic
    • ✅ State updates and transitions
    • ✅ Output based on input — not internal implementation

    It keeps your tests focused, maintainable, and aligned with user expectations.
