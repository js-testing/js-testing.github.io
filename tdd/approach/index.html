    üìò Uncle Bob's Advice on This

    Uncle Bob strongly supports writing tests before refactoring, even in test-last scenarios:

    > "You cannot refactor without tests. If you try, you are just changing code blindly and hoping it still works."
‚ÄîRobert C. Martin

    His book ‚ÄúWorking Effectively with Legacy Code‚Äù (by Michael Feathers, which he endorses) emphasizes this strategy heavily.

    Refactoring Scenario Suggested Testing Approach
    Code already exists ‚úÖ Characterization tests (Test-Last)

    üõ†Ô∏è Uncle‚ÄØBob on Test-First vs Test-Last

    ‚úÖ Test-First (TDD proper)

    • Red/Green/Refactor cycle: write a failing test, then code to pass it, then refactor‚Äîfollowing the Three Laws.

    • Treats tests as executable specifications. They guide the code design and drive modular, decoupled architecture.

    • Emphasis on keeping tests F.I.R.S.T (Fast, Independent, Repeatable, Self-validating, Timely) to maintain rapid feedback.

    • Encourages outside-in design using test doubles (London style) to mock slow parts and focus design.

    • Transformation Priority Premise helps choose simple transformations first, making the implementation process clearer.

    Occurs:

    Before writing any implementation code.

    Steps:

    1. Write a failing test ‚Äî Describe the expected behavior or output of a component or function.
    2. Run the test ‚Äî Confirm that it fails (this ensures the test is valid).
    3. Write the minimal code ‚Äî Implement just enough to make the test pass.
    4. Run the test again ‚Äî It should now pass.
    5. Refactor ‚Äî Clean up the code while keeping tests green.
    Example
    // Test (written first)
test('Button displays correct label', () => {
    render(<Button label="Click me" />);
    expect(screen.getByText('Click me')).toBeInTheDocument();
});

    üß™ Test-Last (not TDD)

    • You write the bulk of production code first; tests come later.

    • Less guidance; code may evolve without early constraints, possibly leading to tightly coupled or untestable designs.

    • Easier to slip into without discipline, but risking long build times and lower test coverage.

    • Tends to miss the benefits of early abstraction, clean architecture, and design-for-testability.

    Occurs:

    After writing implementation code.

    • You write a component or function first.
    • Then, you write tests to cover what you've already implemented.
    • This is not TDD, but still a form of testing (often called test-after or test-last development).
    • üö´ Not TDD, but better than no tests.
    • ‚úÖ Can be useful during legacy refactoring or prototyping.

    üí° Which Approach Should You Choose?

    Scenario Recommendation Why
    Want cleaner design & rapid refactoring Test‚ÄëFirst (TDD) Acts as built-in spec, encourages testable, modular code, and keeps cycle tight and safe.
    Working under time pressure or on exploratory code Test‚ÄëLast (with caution) Offers faster progress but risks technical debt and brittle code if tests delayed too long.
    Maintaining existing legacy code Hybrid Add tests where needed, but gradually adopt TDD for new features.
    User-facing behavior / system flow Use BDD/ATDD Write high-level tests first (e.g. Gherkin), then implement with TDD under the hood ([blog.cleancoder.com][1], [cleancoders.com][2], [arxiv.org][3]).
