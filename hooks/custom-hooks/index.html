    Custom Hooks

    🧠 Types of Custom Hooks (by purpose)

    Hook Type Description Returns Something?
    State management hook Manages internal state and exposes it along with setters ✅ Yes
    Data fetching hook Fetches and returns API data, loading/error state ✅ Yes
    Utility logic hook Encapsulates reusable logic (e.g., debounce, timers, validation) ✅ Often
    Side effect hook Responds to inputs and performs side effects (e.g., logging, state reset) ❌ Not usually
    Event handler hook Exposes memoized event handlers or shortcuts for logic ✅ Often

    🔍 Examples

    1. Returns state/data (common)

    function useCounter(initial = 0) {
  const [count, setCount] = useState(initial);
  const increment = () => setCount((c) => c + 1);
  return { count, increment };
}

    2. Returns nothing — side-effect only (like yours)

    function useLogOnMount(message) {
  useEffect(() => {
    console.log(message);
  }, []);
}

    3. Mixed: optionally returns something

    Some hooks return cleanup functions or trigger callbacks, but still don’t return values directly. This is common with hooks that operate on refs or lifecycle logic.

    🧾 When to return vs not return from a custom hook

    • ✅ Return values when the hook creates/manages state, performs calculations, or provides functions the component needs.
    • ❌ Don't return if the hook only orchestrates side effects (like triggering API calls, form resets, event subscriptions).

    ✅ Summary

    Your hook is a side-effect-only custom hook — it doesn't need to return anything because all the effects are handled through setX functions and form.

    centralized hooks or utilities for form resets

    creating centralized hooks or utilities for form resets is a good practice, especially when:

    • You reuse similar form.setFieldsValue() calls across multiple components.
    • Your form-reset logic becomes complex or spans multiple conditions.

    ✅ Option 1: Centralized Hook for Form Reset

    If you're using React and Ant Design, you can create a custom hook that handles various form reset scenarios.

    // hooks/useResetForexForm.js

export const useResetForexForm = ({ form, FOREX_DEAL }) => {
  const resetCommonFields = () => {
    form.setFieldsValue({
      [FOREX_DEAL.SELLING_AMOUNT]: undefined,
      [FOREX_DEAL.BUYING_AMOUNT]: undefined,
      [FOREX_DEAL.DEAL_APPROVERS]: null,
      [FOREX_DEAL.DEAL_VERIFIERS]: null,
    });
  };

  const resetOnBuy = () => {
    form.setFieldsValue({
      [FOREX_DEAL.SELLING_CURRENCY]: undefined,
    });
    resetCommonFields();
  };

  const resetOnSell = () => {
    form.setFieldsValue({
      [FOREX_DEAL.BUYING_CURRENCY]: undefined,
    });
    resetCommonFields();
  };

  return {
    resetOnBuy,
    resetOnSell,
    resetCommonFields,
  };
};

    📦 How to Use It in a Component or Another Hook

    import { useResetForexForm } from "./hooks/useResetForexForm";

const YourComponent = () => {
  const [form] = Form.useForm();

  const FOREX_DEAL = {
    BUYING_CURRENCY: "buyingCurrency",
    SELLING_CURRENCY: "sellingCurrency",
    SELLING_AMOUNT: "sellingAmount",
    BUYING_AMOUNT: "buyingAmount",
    DEAL_APPROVERS: "dealApprovers",
    DEAL_VERIFIERS: "dealVerifiers",
  };

  const { resetOnBuy, resetOnSell } = useResetForexForm({ form, FOREX_DEAL });

  useEffect(() => {
    if (transactionType === "BUY") {
      resetOnBuy();
    } else if (transactionType === "SELL") {
      resetOnSell();
    }
  }, [transactionType]);

  return <Form form={form}>{/* form fields */}</Form>;
};

    ✅ Benefits of Centralizing Form Reset Logic

    Benefit Description
    💡 Reusability Use the same reset logic across multiple hooks or components
    📦 Maintainability Central place to update field names, null/undefined logic, or conditions
    🧪 Testability Easy to unit test isolated form reset logic if needed
    🔍 Readability Cleaner useEffect blocks — they call a single reset function

    🔁 Optional Enhancements

    You can make the hook more flexible by:

    • Accepting field names as params: resetFields(['field1', 'field2'])
    • Supporting null, undefined, or default values
    • Including validation resets (form.resetFields() optionally)

    ✅ Summary

    Yes — it's a best practice to move form reset logic into a centralized custom hook (like useResetForexForm) when:

    • The reset logic is reused or complex
    • You want cleaner code and better maintainability
